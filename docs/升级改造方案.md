# AI Pulse 升级改造方案

> 版本：v1.0 ｜ 日期：2026-07-15 ｜ 状态：待评审
> 目标：把"RSS 聚合器"升级为**证据驱动、LLM 增强的 AI 行业舆情分析系统**

---

## 0. 背景与目标

当前 AI Pulse 是一个纯 Node.js 静态站（采集 → 去重评分 → SSG → Pages）。经代码走查与运行数据诊断，存在四类问题：数据可靠性崩塌（来源成功率仅 36%）、聚类名不副实、无真正舆情分析、若干功能 Bug 与工程债。

本次升级四档全做，并接入国内大模型做智能分析。总目标：

1. **数据能进来**（成功率 ≥ 85%）
2. **信息有价值**（LLM 摘要/去重归并/真聚类/情感热度/每日速报）
3. **用起来顺手**（搜索/筛选/分页/日报推送）
4. **长期可维护**（配置一致/健康告警/测试/CI）

**运行环境假设**：GitHub Actions（海外）为主，保留 Pages 部署；同时设计国内兜底采集策略（RSSHub/镜像/代理开关），本地在国内也能跑。若与实际不符请指出。

---

## 1. 升级路线图（四阶段）

| 阶段 | 主题 | 优先级 | 交付核心 |
|------|------|--------|----------|
| 阶段一 | 数据可靠性（P0） | 🔴 最高 | 来源体检、修截断 Bug、token、国内外分池、健康门禁 |
| 阶段二 | 智能化分析（LLM） | 🟠 高 | 摘要/去重归并/主题聚类/情感热度/每日速报 |
| 阶段三 | 产品化体验 | 🟡 中 | 真聚类展示、搜索/筛选/分页、RSS 输出、日报推送 |
| 阶段四 | 工程化 | ⚪ 持续 | 配置数据一致、健康看板告警、测试、CI 缓存 |

阶段一必须先落地（否则站点是空的）；阶段二是价值跃迁；三、四增强可用性与可维护性。

## 2. 阶段一：数据可靠性（P0）

### 2.1 问题与对策

| 问题 | 根因 | 对策 |
|------|------|------|
| Tier1 一手源挂 9/13 | RSS 地址失效改版 | 逐源体检，替换为现行有效地址 |
| 大 feed 解析失败 | `maxBytes=512KB` 截断致 XML 不完整 | 提升至 3–5MB，并对截断内容做安全兜底/不解析截断体 |
| 墙外源超时 | 国内网络 + 无兜底 | 国内外分池：超时分级 + RSSHub/镜像兜底 + 代理开关 |
| GitHub API 限流 403 | 未认证 60 次/时 | 读取 `GITHUB_TOKEN`，认证后 5000 次/时 |
| 配置漂移 16↔25 源 | 代码与快照不同步 | 重建统一源清单，采集后校验源数量一致 |

### 2.2 具体改动

- **`src/sources.mjs`**：重建源清单，每源加 `enabled`、`fallbackUrl`(RSSHub/镜像)、`needProxy` 字段；国内源与全球源分组。
- **`src/lib/fetch.mjs`**：`maxBytes` 提升；截断时标记 `truncated` 且不将截断体交给 XML 解析；`BLOCKED_HOSTS` 改为返回 `{ok:false}` 而非 throw（与调用方约定统一）；支持 `HTTP_PROXY` 环境变量。
- **`src/collectors/github.mjs`**：请求头加 `Authorization: Bearer ${GITHUB_TOKEN}`（有则用）。
- **新增 `src/lib/source-check.mjs`**：一键体检脚本，输出每源可用性报告（供人工替换失效源）。
- **`src/collect.mjs`**：采集后写健康门禁——成功率低于阈值时告警（非阻断，记录到 health.json）。

### 2.3 验收标准
- 来源成功率 ≥ 85%
- 大 feed（arXiv/36氪）能正常解析
- 无 token 与有 token 均可运行（有则不限流）

## 3. 阶段二：智能化分析（接入 agnes AI）

### 3.1 LLM 接入设计

- **新增 `src/lib/llm.mjs`**：封装 OpenAI 兼容接口调用（带并发限流、重试、失败降级）。
- **配置（`.env`，已被 .gitignore 忽略）**：

```
LLM_BASE_URL=https://apihub.agnes-ai.com/v1
LLM_MODEL=agnes-2.0-flash
LLM_API_KEY=***（你提供，写入 .env，绝不入库）***
GITHUB_TOKEN=***（可选）***
```

- **新增 `.env.example`**：只放占位符，供团队参考。
- **降级策略**：LLM 不可用时，自动回退到阶段一的纯规则管线，站点仍可构建。

### 3.2 LLM 用在哪些环节

| 环节 | 作用 | 产物 |
|------|------|------|
| 摘要 | 每条信号生成 1–2 句中文摘要 | `signal.aiSummary` |
| 语义去重归并 | 判断"同一事件多来源" | 真正合并 alternativeSources |
| 主题聚类 | 归到 5 主线 + 少量子话题，取代 160+ 噪音标签 | `signal.topics` |
| 情感 / 热度 | 打分 + 倾向（正/中/负） | `signal.sentiment`、`heat` |
| 每日速报 | 汇总当日 Top 变化成一段"今日 AI 要点" | `snapshot.dailyBrief` |

### 3.3 成本控制
- 只对**去重后的信号**调用，且按分数取 Top N（如 120 条）做摘要/情感
- 结果写入 snapshot 并**缓存**（按 signal.id），次日增量只处理新信号
- agnes-2.0-flash 为免费模型，成本可控；限流用队列 + 退避

### 3.4 验收标准
- 首页/主线出现 AI 摘要与每日速报
- 聚类从 160+ 收敛到"5 主线 + 有限子话题"
- 断网/断 key 时构建不崩（走降级）

## 4. 阶段三：产品化体验

- **真聚类展示**：首页/timeline 按语义主题呈现，去掉噪音标签墙。
- **搜索 / 筛选 / 分页**：`dist/data.json` 输出结构化数据，前端 `app.js` 做客户端搜索、按 tier/region/主题筛选、分页；HTML 不再写死全部数据。
- **修复现有 Bug**：`换一个` 按钮绑定事件；lines 页 tab 改为可切换或链接；timeline 多标签筛选修正。
- **RSS / JSON 输出**：生成 `dist/feed.xml`，让 AI Pulse 自己可被订阅。
- **日报推送（可选）**：每日速报推送到微信（企业微信机器人）/ 邮件，需你提供 webhook 或邮箱配置。

### 验收标准
- 站点可搜索、可筛选、可分页
- 死按钮全部修复
- 有可订阅的 feed

---

## 5. 阶段四：工程化

- **配置数据一致性**：采集时校验源清单与快照一致，漂移即告警。
- **来源健康看板**：`signals` 页展示每源真实健康状态（成功/失败/延迟/最后成功时间），失败高亮。
- **测试**：为 `cluster.mjs`、`fetch.mjs`、`llm.mjs` 加最小单测。
- **CI 优化**：`daily-refresh.yml` 加依赖缓存、失败通知、snapshot 产物留存。

## 7. 风险与兜底

| 风险 | 兜底 |
|------|------|
| 部分源永久失效 | 体检报告人工确认后替换，保留 fallbackUrl |
| LLM 限流/不可用 | 自动降级到规则管线，站点照常构建 |
| 国内网络抓不到墙外源 | RSSHub/镜像 + 代理开关，双环境兼容 |
| API key 泄露 | 只存 .env（已忽略），文档与代码不写明文 |

## 8. 建议执行顺序

**阶段一 → 阶段二 → 阶段三 → 阶段四**，每阶段独立可交付、可验证。

阶段一不依赖 LLM，风险最低、价值最高，建议立即开工。确认后我从阶段一动手，每完成一阶段跑一次采集+构建做验证再进入下一阶段。
